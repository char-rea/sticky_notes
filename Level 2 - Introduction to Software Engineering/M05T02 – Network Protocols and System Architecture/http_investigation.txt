--- Method Interaction: ---

Get vs Post:
GET:
- Used to retrieve data from a server.
- Parameters are sent in the URL query string (e.g. ?key=value).
- Data is visible in the URL and often cached or logged.
- Typically does not change server state.
- Limited by URL length.

POST:
- Used to send data to a server for processing.
- Parameters are sent in the request body, not the URL.
- Data is not visible in the address bar.
- Can submit larger and more complex data.
- Commonly used for form submissions and API data creation.

Purpose of the payload:
    The payload represents the data being sent to the server. 
    The payload here is a simple form field: key=value
    It is transmitted in the HTTP request body, not in the URL.
    The server reads this bosy and processes the key-value pairs as form data.
    The Content-Length: 9 header confirms the size of the transmitted payload.
    This allows POST requests to send structured or sensitive data more securely than GET.

Response Body:
{

Status Code: 200 OK confirms the POST request was successful.
The form object echoes back the payload showing "key" : "value" 
"args" is empty, showing that no query-string parameters were used (unlike in the GET)
Headers show metadata about the request, such as user agent and length.

The server reflects the POST(ed) data by returning it in the "form" section of the response body, confirming it was correctly received and parsed.



--- Status Code Exploration ---

1. 404 – Not Found
The server was reached successfully, but the requested resource does not exist at that URL.

Why this might occur
- The endpoint path is incorrect or misspelled.
- The resource has been removed or never existed.
- The API expects a different route or HTTP method.
- The server is configured to deliberately return 404 for unknown paths.

2. 403 - Forbidden 
The server understands the request but refuses to authorise it.

What this tells you about access restrictions
- The endpoint exists, but access is denied.
Common reasons include:
- Missing or invalid authentication credentials
- Insufficient permissions
- IP‑based or security policy restrictions
- Endpoint designed to always return a specific status code (as with /status/403)

3. Key Headers

Header  -	Purpose
content-type: application/json; charset=utf-8   -	Indicates the error response body is JSON‑formatted
content-length: 14	-   Shows a minimal response body (likely a short error message)
etag    -	Identifies the specific version of the error response
server: cloudflare	   -   Indicates the response passed through Cloudflare
cf-cache-status: DYNAMIC    -	Confirms the error was generated dynamically, not cached
x-envoy-upstream-service-time   -	Shows how long the upstream service took to respond
cf-ray  -	Cloudflare request ID used for tracing and diagnostics
set-cookie  -	Session and security cookies set even during error responses

Errors are communicated primarily through the HTTP status code, not the body.
The headers still include full metadata, showing:
The request was processed normally at the protocol level
Security and session handling still applies, even on errors
